# CIH Wiki Improvement — 10-Expert Panel Review & Roadmap

_Date: 2026-07-09. Produced by a 10-agent expert panel review of the wiki subsystem
(`crates/cih-wiki`, `crates/cih-engine/src/wiki/`, `crates/cih-engine/src/llm/`,
`docs-viewer/`), benchmarked against OpenWiki (LangChain's LLM documentation agent)._

## Context

The `cih-wiki` crate (~4.4k lines) renders the CIH code graph into a role-based
(PO/BA/Dev) documentation wiki. Compared against OpenWiki, CIH's wiki is structurally
accurate and scales to 50k-file legacy repos, but lacks a narrative "why" layer, a
freshness/update loop, prose quality, and distribution to its highest-value audience
(coding agents). Ten expert reviewers critiqued the pipeline from different angles;
their findings and a synthesized roadmap follow.

---

## Expert panel findings

### Expert 1 — Information architect
**Weaknesses:** (1) Feature hub pages emit BROKEN relative links: feature_index.rs:44-60 renders `{feature}/po.md` from a page already at `pages/{feature}/index.md` → resolves to `/{feature}/{feature}/po.md` 404; dev links use build_dev_page_paths while files are named by assign_class_slugs — two independent slug systems with separate collision counters that diverge; only one dev page per community linked at all. (2) Dev pages are dead ends: zero markdown links; callers/callees/tests/tables are backtick text; cross-linking one-way (api_flow→dev only). (3) communities/ is a redundant orphaned hierarchy in default package mode — Pkg: synthetic nodes duplicate each feature 1:1, unbounded member tables, false "detected by Leiden" caption, linked from nowhere. (4) Persona buttons broken: gen-index.js needs nav['system'] which emit_global_pages never populates; pages/po.rs + ba.rs renderers are dead code — "Product Owner" button lands on alphabetically-first feature. (5) No scale strategy: autogenerated sidebar → 500 flat items per feature; routes.md flat table with no links either direction.
**Proposals:** central LinkContext resolver + link-integrity fixture test (M/HIGH); bidirectional dev links via existing class_dev_slugs (S/HIGH); suppress communities/ in package mode, integrate in graph mode, delete dead po.rs/ba.rs (S/MED-HIGH); generate system-level po/ba rollups + wire nav['system'] (S-M/HIGH); stereotype sub-folders + dev index + capped member tables (M/MED).

### Expert 2 — Technical writer
**Weaknesses:** (1) Graph-only (default) "Business Overview" pages contain zero sentences — stat line + pipe tables (feature_po.rs:154); BA pages show numbered method names + file:line to business analysts. (2) llm-full concatenates N isolated per-community po_summary paragraphs under one heading (feature_po.rs:50-98) — no synthesis unless feature_llm path fires; fallback silent. (3) Prompts (llm/prompts.rs) never ask for "why"/design rationale/change guidance; git history entirely unused across wiki code; Javadoc stripped before class-enrich LLM sees bodies (class_enrich.rs:99 uses b.stripped); CLASS_SYSTEM_PROMPT bans class names → renderer launders output (clean_method_desc). (4) Evidence budgets starve prose: MAX_EVIDENCE_CHARS=3000 (evidence.rs:9), MAX_FEATURE_EVIDENCE=6000 for a whole feature. (5) Role banners promise stakeholder view even when all narrative silently dropped. (6) step_descriptions.get(i) index-aligned in 3 renderers — wrong count → prose attaches to wrong steps.
**Proposals:** deterministic lead-paragraph NLG for graph-only (S/HIGH); make feature-level synthesis mandatory, kill concatenation (M/HIGH); feed git log subjects + unstripped Javadoc into evidence packs, add design_rationale + change_guidance prompt fields (M/HIGH); honest degradation admonition when LLM absent (S/MED); raise/configure evidence caps ~24k + key step descriptions by method id (S/MED).

### Expert 3 — Graph/static-analysis
**Weaknesses:** (1) `index_edges` (cih-wiki/src/graph.rs:352-418) silently drops Extends, Implements, HasField, Accesses, Uses, MethodOverrides, IntegrationLink, SimilarTo, TaintFlow — no type hierarchy/fields/duplication/cross-repo links on any page. (2) `build_call_chain` follows MethodImplements but not MethodOverrides → abstract-base/template-method chains dead-end; per-step db_access makes unreachable repos' tables invisible. (3) BFS union rendered as linear "Step 1..N" — mutually exclusive dispatch branches shown as one sequence; LLM step_descriptions aligned by position → silent misalignment on drift. (4) Mermaid truncation (8 participants, first-appearance order) drops the repo/DB layer first; depth hardcoded 4; arrow dedup keeps only first method label per class pair. (5) Edge `confidence`/`reason` discarded — speculative edges rendered as fact. (6) cih-taint output (`.cih/artifacts-taint/`), SimilarTo, untested_paths, entrypoint_kind metadata never reach any page.
**Proposals:** follow MethodOverrides (S/HIGH); render dispatch branches as mermaid `alt` + "one of:" lists (M/HIGH); sink-prioritized truncation + configurable caps (S-M/HIGH); taint/security section on API-flow pages from artifacts-taint (M/HIGH — differentiator); confidence-aware "(inferred)" marking (S/MED); dev-page type hierarchy/fields/duplicates from dropped edges (M/MED).

### Expert 4 — LLM/RAG engineer
**Weaknesses (several are outright bugs):** (1) Cache key = evidence hash only (feature_enrich.rs:331, flow_enrich.rs:213, class_enrich.rs:116) — switching provider/model/language/prompt silently serves stale cached output. (2) Failed class enrichments cached as permanently-empty entries with valid hash (class_enrich.rs:189-193) — transient rate-limit poisons cache until source changes. (3) llm-full community tier NEVER cached: run.rs:593 passes empty community_updates; WikiModuleCacheEntry can't hold CommunityLlmFull — most expensive tier re-paid every run. (4) Evidence starvation: 3k chars/community, 3 snippets × 10 lines taken from file at first method's start line (not method bodies); feature cap 6k breaks after ~C3 while citation map still lists dropped IDs; paid-for class summaries never fed upward. (5) Citations mostly dead: only snippet items get URLs; no verifier — hallucinated [C3-R9] renders as-is. (6) llm-summary mode duplicates the same string into po and ba (class_enrich.rs:246-259) — personas cosmetic. (7) `&body[..600]` byte-slice panic on multibyte (class_enrich.rs:284). (8) Adapters: no 429/Retry-After handling, no token/cost reporting (est. $25-80 per 50k-file llm-full run, invisible), timeout thread leak, brace-scraping instead of structured output.
**Proposals:** composite cache key (evidence‖model‖lang‖PROMPT_VERSION‖mode) + retryable failure markers (S/HIGH); cache llm-full tier (M/HIGH); hierarchical map-reduce — class summaries feed community prompts, community feeds feature; token-budgeted packing (M-L/HIGH); citation verifier + full citation map (S-M/MED); adapter hardening + usage into WikiLlmInfo + native structured output (M/MED); fix UTF-8 panic, distinct PO/BA reduce, loud fallback tier in wiki_meta (S/MED).

### Expert 5 — DevEx/CI engineer
**Weaknesses:** (1) No git-head no-op gate though every piece exists: cih_core::git_head() (registry.rs:99-107), registry last_git_head, WikiMeta.repo_commit — but repo_commit only ever written None (llm/grouping.rs:506) and run_wiki never reads it. (2) --incremental is LLM-cache-only; generate_wiki at run.rs:581 unconditionally re-renders all pages; WikiModuleCacheEntry.content_hash written as "" forever (cache.rs:31) — skip-if-unchanged designed but never wired; full rewrites churn mtimes (defeats Docusaurus cache, rsync, git review). (3) detect_changes/git_changed_files stranded in cih-server (changes.rs:11, symbol.rs:87) — no `wiki --since <ref>`. (4) Staleness invisible: 3-week-old artifacts render silently; no "graph N commits behind" warning; CI can't verify freshness. (5) analyze→discover→wiki with no orchestrator, no --watch, no CI workflow example, no PR-preview story.
**Proposals:** stamp HEAD into WikiMeta + no-op gate + `wiki --check` exit 2 (S/HIGH, ~1 day); `cih-engine refresh <repo>` orchestrator with per-stage fingerprint skip (M/HIGH); write-if-different rendering then `wiki --since` diff-scoped regen via file_dev_map (M→L/HIGH); staleness warning vs registry head (S/MED); ship docs/ci/wiki.yml + PR-bot example + incremental-behavior test (S/MED).

### Expert 6 — Coding-agent consumer
**Weaknesses:** (1) Wiki invisible to agents: no MCP resource/tool serves wiki markdown (resources.rs:25-53 = 4 graph resources); nothing writes an AGENTS.md/CLAUDE.md pointer into target repos. (2) manifest.json can't index for agents: PageEntry lacks FQN/source-file mapping; dev pages emitted community_id:None, title=simple class name — file→page lookup impossible though page bodies carry rich file:line refs. (3) Slugs unstable across regens: adding a 2nd same-named class RENAMES the first page (base → base-1); build_dev_page_paths suffix order depends on community_ids iteration → Leiden re-runs reshuffle URLs; cached citations/links rot. (4) No change-guidance section though graph knows callers/tests/shared tables. (5) Silent enrichment downgrade unrecorded per-page — agent can't distinguish LLM prose from template boilerplate.
**Proposals:** MCP wiki resources cih://repo/{name}/wiki/{slug} + ask_wiki tool (M/HIGH); AGENTS.md pointer + agent-index.json {file→pages, fqn→page} at end of run_wiki (S/HIGH); FQN-hash stable slugs, never retro-rename (S/HIGH); "Modifying this class" section from calls_in/Tests/tables (M/HIGH); per-page provenance front matter (enrichment tier, generated_at, source_files, graph_version) (S/MED).

### Expert 7 — Rust systems engineer
**Weaknesses:** (1) New page type touches ~6 stringly-typed places; no page abstraction; registration boilerplate ×10 in emit_* blocks (lib.rs:316-904); manifest roles vec disagrees between lib.rs:1155 and 1311; generate_wiki = 430-line monolith. (2) Renderers fs::write mid-render — no in-memory RenderedPage to hash/diff, which is exactly where render-incrementality would plug in. (3) 25 #[allow(too_many_arguments)] workspace-wide; WikiInput 22 fields/9 Options; grouping passed as untyped String though engine has WikiGrouping enum (erased at run.rs:568). (4) Method-id→class-id heuristic copy-pasted 3×; enrich_route_flows duplicated in both if-let branches (run.rs:458-510). (5) Every Node cloned 4-6× across WikiGraph maps — gigabytes at Fineract scale. (6) NO markdown/MDX escaping anywhere: `|` in symbol names breaks tables, `<`/`{` in LLM prose breaks Docusaurus MDX compilation — whole-build failure class. (7) schema_version=1 theater; validate_module_tree hard-rejects anything else, no migration.
**Proposals:** PageRenderer trait + PageSink with in-memory RenderedPage + content hashing — DO FIRST, unlocks new page types (DB/DDL, coverage), rayon rendering, and incrementality hook (M/HIGH); MdWriter with escape_cell()/mdx_safe() (S/HIGH); typed boundary — WikiGrouping into input, split WikiInput into GraphData/Enrichments/RenderOptions, single symbol_ids module (S/MED); borrowed/interned WikiGraph (M/MED-HIGH at scale).

### Expert 8 — Legacy Java/Spring architect
**Weaknesses:** (1) PackageStrategy::classify (cih-grouping/strategies/package.rs:27-88): Strategy 3 nested inside Strategy-2 branch — Ant-era `src/com/bank/...` layouts send EVERYTHING to "shared"; even when it fires, deepest-segment yields ~300 flat leaf-features on Fineract (no domain hierarchy); wiki-side fallback guesses silently. (2) @Transactional invisible — zero handling workspace-wide; TX boundaries (the #1 Spring debugging question) unanswerable. (3) No config-property capture (@Value/@ConfigurationProperties) — "what flag changes this behavior?" has no answer. (4) DB access forward-only — no per-table reverse index ("which job writes m_loan_transaction?"); data already in ReadsTable/WritesTable edges. (5) Dev pages omit incoming callers + inheritance (calls_in/Extends/Implements exist, unrendered) — "what breaks if I change this entity?" unanswerable. (6) API-flow pages labeled role:po despite being call-chain tables; no domain/scenario lifecycle pages; dev index dumps 40 lines of unresolved-refs.md as first impression.
**Proposals:** un-nest Strategy 3 + two-level domain/subfeature hierarchy (S/HIGH — gates everything); @Transactional as node prop + TX markers in flow tables (M/HIGH); per-table pages, reverse index (S-M/HIGH); config-property index page (M/MED-HIGH); "Used by" + inheritance block on class pages (S/HIGH); domain scenario pages, relabel api-flow role:ba (M-L/MED).

### Expert 9 — Product manager
**Weaknesses:** (1) Time-to-first-wiki ≈ 7 commands across 3 toolchains (setup.sh → .env → compose → analyze → discover → wiki → docs-viewer npm start); wiki hard-fails without discover. (2) 25 wiki flags (args.rs:427-525), and defaults (graph mode, package grouping) produce the flattest output — first impression is the weakest tier, degradation silent. (3) Differentiator unstated + undistributed: README leads with "MCP server"; no MCP wiki tool/resource; no AGENTS.md pointer written into target repos (OpenWiki does both). (4) No freshness loop: --incremental only caches LLM; render full-regen, not git-aware despite detect_changes existing. (5) Roadmap prioritizes expansion (chat frontend, multi-repo, topology viewer) over broken core loop; Phase 23 marked "no further work needed".
**Proposals (ranked):** 1. `cih-engine docs <repo>` one-command analyze→discover→wiki→serve (M/H). 2. Git-aware freshness: HEAD in wiki_meta, no-op check, feature-scoped re-render via detect_changes, `wiki --check` for CI (M/H). 3. MCP wiki resource + get_wiki_page tool + AGENTS.md emission (S/H). 4. Collapse flags: auto provider from env key, default llm-summary when key exists, auto grouping, loud fallback (S/M-H). 5. Roadmap reorder: defer topology/multi-repo, keep DB/DDL pages, reopen Phase 23 (S/M).
**Positioning:** own "the wiki that's never wrong" — deterministic, evidence-grounded docs agents can cite.

### Expert 10 — Security/ops reviewer
**Weaknesses:** (1) No secret redaction anywhere: bodies.rs strips only noise lines — hardcoded JDBC passwords/API keys/PEM blocks flow verbatim into dev pages, the single-file html viewer (which embeds every page), and evidence/*.json; zero "redact" hits in cih-wiki. (2) SECURITY.md never mentions wiki, yet `wiki --llm` sends source snippets + 600-char bodies + BRD chunks to external providers; no endpoint allowlist / on-prem mode. (3) docs-viewer binds 0.0.0.0:3001 with no auth (package.json start scripts); CIH_WIKI_REPOS_DIR serves every repo's source-bearing wiki to the network (contrast cih-server which refuses non-loopback without token). (4) LLM-returned feature names used raw in write paths (features.rs:32-41 fast-path → out_dir.join(...)) — `"../../x"` writes outside wiki dir; slugify exists but isn't applied; reads are guarded (safe_repo_path), writes aren't. (5) Failed enrichments permanently cached (confirms E4). (6) No atomic output (mid-run crash = mixed old/new wiki served), no run cost/time budget, no gitignore check for .cih/.
**Proposals:** redaction pass at bodies.rs output + EvidencePack::render (M/HIGH); SECURITY.md wiki-egress section + wiki.llm_allowed_hosts (S/HIGH); safe_page_path() slugified write segments + charset check in validate_module_node (S/MED); never cache failures + end-of-run failure summary (S/HIGH); stage-and-swap to wiki.tmp + atomic rename (M/MED); docs-viewer default 127.0.0.1 + gitignore warning (S/MED).

# SYNTHESIS

## Consensus themes (independent experts converged)

1. **Silent degradation is systemic** (E1, E2, E3, E4, E5, E6, E10): LLM fallback tiers, permanently-cached failures, stale caches after model/prompt changes, mermaid truncation, dropped edges, broken links, stale artifacts — none of it warns the reader or the operator. The wiki's core value proposition is trustworthiness; silence is the #1 threat to it.
2. **The graph computes far more than the wiki renders** (E3, E8): inheritance, incoming callers, MethodOverrides dispatch, taint flows, edge confidence, per-table reverse indexes, TX-relevant structure — all indexed upstream, all invisible on pages. Cheapest quality wins in the whole review.
3. **Freshness machinery exists but was never wired** (E5, E9): git_head(), registry last_git_head, WikiMeta.repo_commit (always None), content_hash (always ""), detect_changes (stranded in cih-server). OpenWiki's entire maintenance loop is reachable with plumbing, not research.
4. **Distribution gap** (E6, E9): the wiki is invisible to its highest-value audience (coding agents) — no MCP surface, no AGENTS.md pointer, no agent-index; and the human onboarding is ~7 commands.
5. **Real bugs, not just design debt**: broken feature-hub links + diverging slug systems (E1), UTF-8 panic (E4), cache poisoning (E4/E10), uncached llm-full tier (E4), no MDX escaping = build-breaking (E7), slug renames on regen (E6), LLM path traversal (E10), PO==BA duplication (E4).
